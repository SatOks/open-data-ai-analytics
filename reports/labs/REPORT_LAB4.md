# Лабораторна робота №4

## Тема: Інфраструктура як код. Розгортання Docker-проєкту в Microsoft Azure

**Дата виконання:** 28 квітня 2026 р.

---

## Мета роботи

Навчитися:
- працювати з хмарною інфраструктурою без локального середовища розгортання
- використовувати Azure CLI як основний інструмент управління Azure
- створювати ресурси Azure за допомогою скриптів
- автоматизувати початкове налаштування Linux VM через cloud-init
- розгортати контейнеризований застосунок у хмарі

---

## Виконані завдання

### 1. Підготовка інфраструктурних скриптів

Створено директорію `infra/` з наступними файлами:

| Файл | Призначення |
|------|-------------|
| `cloud-init.yaml` | Автоматична конфігурація VM при першому запуску |
| `azure-deploy.ps1` | PowerShell скрипт розгортання (Windows) |
| `azure-deploy.sh` | Bash скрипт розгортання (Linux/macOS) |
| `azure-destroy.ps1` | PowerShell скрипт видалення ресурсів |
| `azure-destroy.sh` | Bash скрипт видалення ресурсів |

### 2. Ресурси Azure

Скрипт створює наступні ресурси:

```
Azure Resource Group (odaa-rg)
├── Virtual Network (odaa-vnet)
│   └── Subnet (odaa-subnet, 10.0.1.0/24)
├── Network Security Group (odaa-nsg)
│   ├── AllowSSH (port 22)
│   ├── AllowWeb (port 8080)
│   ├── AllowGrafana (port 3000)
│   └── AllowPrometheus (port 9090)
├── Public IP Address (odaa-public-ip)
├── Network Interface (odaa-nic)
└── Linux Virtual Machine (odaa-vm)
    ├── Image: Ubuntu 22.04 LTS
    ├── Size: Standard_B1s
    └── Custom Data: cloud-init.yaml
```

### 3. Cloud-init конфігурація

При першому запуску VM автоматично виконується:

1. Оновлення системних пакетів
2. Встановлення Docker та Docker Compose plugin
3. Додавання користувача до групи docker
4. Клонування репозиторію з GitHub
5. Створення необхідних директорій
6. Запуск основного Docker Compose стеку
7. Запуск стеку моніторингу

### 4. Команди розгортання

```powershell
# Вхід в Azure
az login

# Розгортання
cd infra
.\azure-deploy.ps1 -ResourceGroup "odaa-rg" -Location "eastus"

# Перевірка стану VM
az vm show --name odaa-vm --resource-group odaa-rg --query "provisioningState"

# Перевірка cloud-init
ssh azureuser@<PUBLIC_IP> 'cloud-init status'

# Перевірка застосунку
curl http://<PUBLIC_IP>:8080/health

# Видалення ресурсів
.\azure-destroy.ps1 -ResourceGroup "odaa-rg"
```

## Висновки

У ході виконання лабораторної роботи було:

1. **Вивчено Azure CLI** — основні команди для створення ресурсів (resource group, vnet, vm, nsg)
2. **Створено інфраструктуру як код** — скрипти PowerShell та Bash для автоматизованого розгортання
3. **Використано cloud-init** — автоматичне налаштування VM при першому запуску
4. **Розгорнуто застосунок** — контейнеризований проєкт працює в Azure без ручного втручання
5. **Забезпечено доступність** — веб-інтерфейс доступний через публічну IP-адресу

Перевага підходу IaC (Infrastructure as Code):
- Відтворюваність — можна створити ідентичне середовище будь-коли
- Версіонування — скрипти зберігаються в Git
- Автоматизація — мінімум ручних дій
- Документація — код сам є документацією інфраструктури

## Труднощі та їх вирішення

1. **Очікування cloud-init** — після створення VM потрібно 5-10 хвилин на виконання cloud-init. Вирішення: використання команди `cloud-init status` для перевірки.

2. **SSH ключі** — Azure автоматично генерує SSH ключі при використанні `--generate-ssh-keys`. Ключі зберігаються локально.

3. **Порти фаєрвола** — необхідно явно відкривати порти в NSG. Вирішення: створення правил AllowSSH, AllowWeb, AllowGrafana, AllowPrometheus.
