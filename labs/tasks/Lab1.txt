Як працює Git
Розуміння внутрішньої будови Git та його сховищ (storage locations).  ￼

На початку важливо визначити, де зберігається наш код. Поширене припущення — що існують лише два місця: одне на віддаленому сервері (наприклад, GitHub), і друге — на нашому локальному комп’ютері. Однак це не зовсім так. Git підтримує три локальні сховища на нашій машині, а це означає, що наш код може бути в чотирьох місцях:  ￼
    •    Робочий каталог (Working directory): місце, де ми редагуємо файли.  ￼
    •    Область індексації / підготовки (Staging area): тимчасове місце, де файли зберігаються для наступного коміту.  ￼
    •    Локальний репозиторій (Local repository): містить код, який уже був закомічений (commit).  ￼
    •    Віддалений репозиторій (Remote repository): віддалений сервер, де зберігається код.  ￼

Більшість Git-команд, по суті, переміщують файли між цими чотирма локаціями. 

Команди Git

Шпаргалка з Git-команд
Зручний довідник найважливіших Git-команд для розробників.  ￼

Початок роботи (Getting Started)
    •    git init — ініціалізує новий Git-репозиторій.  ￼
    •    git clone [url] — клонує репозиторій з віддаленої URL-адреси.  ￼

Внесення змін (Making Changes)
    •    git add [file] — додає файл у staging area (індекс).  ￼
    •    git commit -m "[message]" — фіксує (commit) зміни з описовим повідомленням.  ￼
    •    git status — показує стан робочого каталогу (що змінено/проіндексовано тощо).  ￼
    •    git diff — показує відмінності між робочим каталогом і staging area.  ￼

Гілки та злиття (Branching and Merging)
    •    git branch — показує всі локальні гілки.  ￼
    •    git branch [branch-name] — створює нову гілку.  ￼
    •    git checkout [branch-name] — перемикається на вказану гілку.  ￼
    •    git merge [branch-name] — зливає вказану гілку в поточну.  ￼
    •    git branch -d [branch-name] — видаляє вказану гілку.  ￼

Віддалені репозиторії (Remote Repositories)
    •    git remote add origin [url] — додає віддалений репозиторій.  ￼
    •    git push origin [branch-name] — надсилає зміни у віддалений репозиторій.  ￼
    •    git pull origin [branch-name] — забирає зміни з віддаленого репозиторію.  ￼
    •    git fetch — завантажує зміни з віддаленого репозиторію без злиття.  ￼

Скасування змін (Undoing Changes)
    •    git reset [file] — прибирає файл зі staging area (розіндексовує).  ￼
    •    git checkout -- [file] — відкидає зміни у файлі.  ￼
•    git checkout -- [file] сучасніший варіант git restore [file]).
    •    git revert [commit] — створює новий commit, який скасовує зміни з вказаного commit.  ￼

Завдання 
В результаті роботи ви повинні створити проєкт що складається із наступних модулів:
- завантаження даних (data_load)

- перевірка даних (data_quality_analysis)

- дослідження даних (data_research)

- візуалізація (visualization)

Для розробки кожного модуля створюється окрема гілка, що в кінці розробки об'єднується із гілкою  "main".

Структура репозиторію

Хід роботи
    1.    Створити репозиторій open-data-ai-analytics.
    2.    Додати структуру:
           README.md
           .gitignore
           data/README.md
           notebooks/
           src/
           reports/figures/

    3.    Налаштувати .gitignore для:
        •    __pycache__/, .ipynb_checkpoints/
        •    .venv/, .env
        •    data/raw/ (або будь-якої папки з великими файлами)

        Зробити коміт

    4.    Заповнити README.md:
        •    назва і мета
        •    джерело відкритих даних (посилання)
        •    3 питання/гіпотези для аналізу

        Зробити коміт
      
    5.    Створити гілку feature/data_load і додати скрипт завантаження даних
    6.    Змерджити feature/data_load у main.

    7.    Створити 2 feature-гілки:
    •    feature/data_quality_analysis - додати код для перевірки якості даніх
    •    feature/data_research - додати код для аналізу даних, побудови моделей
    8.    Змерджити обидві гілки у main через PR/MR:
    •    опис “що зроблено”    
    9.    Створити merge-конфлікт у README.md (дві гілки змінюють одну секцію) і розв’язати його коректно.

    10.    Створити гілку feature/visualization і додати код візуалізації даних

    11.    Додати CHANGELOG.md і створити тег v0.1.0.
